## 1. מפרט החומרה ותקצוב המודל (משימה 5.1)

נוצר אוטומטית ע״י `python -m src.hardware` ⟵ `results/hardware.json`.
הדביקו כאן את הפלט האמיתי שלכם:

| שדה | ערך |
|---|---|
| מערכת הפעלה | _לדוגמה_ Windows 11 |
| מעבד (CPU) | _לדוגמה_ Intel i7-12700H (14 ליבות) |
| RAM | _לדוגמה_ 16 GB |
| GPU / VRAM | _לדוגמה_ RTX 3060 / 6 GB (או ללא) |
| מקום פנוי בדיסק / סוג | _לדוגמה_ 380 GB פנוי, NVMe SSD |
| Python | 3.10–3.12 |

**המודל שנבחר:** `Qwen/Qwen2.5-14B-Instruct` (דוגמה).
**הנמקה:** גדול מספיק כדי שטעינה נאיבית מלאה תיכשל על 16GB RAM (כך שצוואר
הבקבוק אמיתי וניתן להדגמה), אך השכבות שלו נכנסות ל-NVMe ושכבת טרנספורמר בודדת
נכנסת ל-RAM — בדיוק האזור שבו AirLLM משתלם. הפונקציה `src.model_selector` ממליצה
על דרגת מודל לפי ה-RAM והדיסק שזוהו; **נמקו את הבחירה הסופית בדוח.**

## 2. מה הניסוי עושה

1. **קו בסיס (משימה 5.2)** — טעינה נאיבית
   (`transformers.AutoModelForCausalLM`, כל המשקולות). על מודל "גדול מספיק" זה
   נכשל ב-OOM / מציף את ה-swap / איטי באופן בלתי-נסבל. הכישלון הזה הוא הראיה
   לצוואר הבקבוק.
2. **AirLLM + קוונטיזציה (משימה 5.3)** — אותו פרומפט דרך AirLLM, שמזרים **שכבה
   אחת בכל פעם** מהדיסק, בסריקה על **FP16 → Q8 → Q4**.
3. **מדידה (משימה 5.4)** — TTFT, ITL/TPOT, תפוקה, שיא RAM/VRAM, אנרגיה.
4. **כלכלה (משימה 5.5)** — On-Prem (CAPEX/OPEX) מול API חיצוני, עם תרחישי
   ענן-GPU ו-Prompt Caching אופציונליים; מציאת נקודת האיזון.

כלי מדידה: חוט רקע `MemorySampler` (שיא RSS + שיא CUDA), טיימר זרימה ל-TTFT
ולפערים בין-טוקנים, ו-`matplotlib` לכל הגרפים.

## 4. ניתוח מושגי ההרצה (משימה 5.6)

- **Prefill = TTFT.** בניית ה-KV-cache על הפרומפט היא כפל-מטריצות מקבילי גדול →
  **חסום-חישוב (compute-bound)**. עקבו אחרי גידול ב-TTFT עם הפרומפט הארוך.
- **Decode = TPOT.** כל טוקן חדש מזרים את כל משקולות המודל דרך הזיכרון פעם אחת →
  **חסום-זיכרון (memory-bound)**. ה-TPOT נשלט ע״י רוחב פס הזיכרון, ולכן קריאות
  הדיסק של AirLLM פוגעות כאן הכי הרבה.
- **AirLLM ≈ זיכרון וירטואלי / Paging** של מערכת הפעלה, אבל עבור *שכבות מודל*:
  רק ה"דף" (השכבה) הנחוץ נמצא בזיכרון, השאר על הדיסק. ה-`Disk I/O` הופך לצוואר
  הבקבוק האמיתי, לא גודל ה-RAM.
- **קוונטיזציה** מקטינה כל משקל (FP16→Q4 ≈ פי 4 קטן יותר), מצמצמת את הבייטים
  שמועברים לכל טוקן → דיקוד מהיר יותר וזיכרון שיא נמוך יותר, עד שהדיוק נפגע.

## 5. ניתוח כלכלי והמלצה (משימה 5.5)

![נקודת איזון](figures/break_even.png)

שני מודלי עלות שקופים ב-[`analysis/economics.py`](analysis/economics.py), כל
ההנחות ניתנות לעריכה:

- **API:** `בקשות × (קלט·מחיר_קלט + פלט·מחיר_פלט)`, עם הנחת **Prompt Caching**
  אופציונלית על הקידומת החוזרת (ספקים בסגנון PagedAttention גובים פחות בהרבה על
  טוקנים מומכנים — זה *מזיז את נקודת האיזון*).
- **On-Prem:** CAPEX מופחת + חשמל (מתוך Wh/בקשה שנמדד) + תחזוקה.
- **ענן GPU (אופציונלי):** תעריף שעתי × שניות/בקשה.

**ממצא לדוגמה:** נקודת איזון ≈ **230 אלף בקשות** בתקופה. מתחת לכך ה-API זול יותר;
מעליה On-Prem מנצח — *לפני* שקלול פרטיות, אבטחת מידע וזמינות לא-מקוונת, שעשויים
להטות לטובת On-Prem ללא קשר לנפח. **ציינו את המחירים האמיתיים וצטטו את הספק.**

## שחזור

```bash
# 1. סביבה (uv בלבד)
uv venv && source .venv/bin/activate      # Windows: .venv\Scripts\activate
uv pip install -e .

# 2. תיעוד חומרה + צפייה במודל המומלץ
python -m src.hardware
python -m src.model_selector

# 3. (אופציונלי) קביעת כונן מהיר שאינו כונן מערכת ההפעלה לשכבות
#    Windows:  set AIRLLM_SHARDS=D:\airllm_shards
#    Linux:    export AIRLLM_SHARDS=/fast/airllm_shards

# 4. הרצת הניסויים (קודם baseline, אחר כך סריקת AirLLM)
python -m experiments.run_baseline
python -m experiments.run_airllm

# 5. בניית טבלאות + גרפים + כלכלה
python -m analysis.analyze
```

### מלכודות ידועות (מתוך רשימת ה-Do/Don't של ההגדרה)
- **Python 3.13 חדש מדי** ל-AirLLM/bitsandbytes — הצמידו 3.10–3.12.
- **כוונו את `layer_shards_saving_path` לכונן מהיר שאינו כונן ההפעלה** — השכבות
  הן עשרות GB של SafeTensors שמציפות את `C:` אחרת.
- השתמשו ב-**`airllm.AutoModel`** (המחלקה הכללית) כדי להימנע משגיאת אי-התאמת
  מחלקה במודלים ממשפחת `Qwen`.
- **לעולם אל תשמרו את ה-Token של Hugging Face** בקוד — הוא מוחרג ב-git כאן.
- התחילו קטן (Q4, `max_new_tokens` נמוך) כדי לוודא שה"צינור" עובד, ואז הגדילו.
