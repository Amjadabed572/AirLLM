## 1. מפרט החומרה ובחירת המודל (משימה 5.1)

זוהה אוטומטית ע״י `python -m src.hardware` ⟵ [`results/hardware.json`](results/hardware.json):

| שדה | ערך (המכונה הזו) |
|---|---|
| מערכת הפעלה | Windows 10 |
| מעבד (CPU) | Intel Core i7-7500U @ 2.70GHz — **2 ליבות פיזיות / 4 לוגיות** |
| RAM | **7.9 GB** |
| GPU / VRAM | NVIDIA GeForce GTX 950M / **2.0 GB** (Maxwell, compute 5.0) |
| דיסק | **SSD** (Micron 1100 SATA), ~40 GB פנוי |
| Python | 3.12 (מנוהל ע״י uv; מוצמד `>=3.10,<3.13`) |

**המודל שנבחר:** `Qwen/Qwen2.5-7B-Instruct`.
**מדוע זו הבחירה הנכונה ("גדול מספיק כדי לכאוב, לא בלתי-אפשרי") *למכונה הזו*:**
טביעת הרגל ב-FP16 (~15GB) **גדולה בהרבה מ-7.9GB RAM**, כך שקו הבסיס הנאיבי
מובטח להיכשל (צוואר הבקבוק אמיתי וניתן להדגמה). עם זאת, שכבות המודל נכנסות
ל-SSD בגודל ~40GB ושכבת טרנספורמר בודדת נכנסת ל-RAM — בדיוק האזור שבו AirLLM
משתלם. מודל 14B/32B לא ייכנס לדיסק הפנוי לאחר כתיבת השכבות; 72B רחוק מהישג יד.
`src.model_selector` משחזר את ההיגיון הזה מה-RAM והדיסק הפנוי שזוהו.

## 2. מה הניסוי עושה

1. **קו בסיס (משימה 5.2)** — טעינה נאיבית
   (`transformers.AutoModelForCausalLM`, כל המשקולות ב-FP16). על 7.9GB RAM זה
   נכשל ב-OOM / מציף swap. **הכישלון הזה הוא הראיה לצוואר הבקבוק**, ונשמר מילולית
   ב-`results/baseline_*.json`.
2. **AirLLM (משימה 5.3)** — אותו פרומפט דרך AirLLM, שמזרים **שכבת טרנספורמר אחת
   בכל פעם** מה-SSD. זוהי הדגמת זרימת השכבות ב-FP16 שמאפשרת הרצה של מודל שאחרת
   בלתי-אפשרי.
3. **השוואת קוונטיזציה (משימה 5.3)** — **Q4 מול Q8 ב-GGUF דרך Ollama**
   (llama.cpp, CPU). *מדוע לא q4/q8 של AirLLM עצמו?* המסלול הזה דורש `bitsandbytes`
   עם GPU של NVIDIA בעל compute capability ≥ 7.5 וכמה GB של VRAM; ה-GTX 950M
   (Maxwell 5.0, 2GB) אינו יכול להריץ זאת — מגבלת חומרה מתועדת. GGUF על CPU נותן
   את מספרי הקוונטיזציה האמיתיים במקום.
4. **מדידה (משימה 5.4)** — TTFT, ITL/TPOT, תפוקה, שיא RAM/VRAM, אנרגיה.
5. **כלכלה (משימה 5.5)** — On-Prem (CAPEX/OPEX) מול API חיצוני, עם תרחישי
   ענן-GPU ו-Prompt Caching אופציונליים; חישוב נקודת האיזון.

כלי מדידה: חוט רקע `MemorySampler` (שיא RSS + שיא CUDA), טיימר זרימה ל-TTFT
ולפערים בין-טוקנים, מוני הזמן המקוריים של Ollama (דיוק ננו-שנייה) להרצות GGUF,
ו-`matplotlib` לכל הגרפים.

## 3. סיכום תוצאות (נמדד)

`airllm-bench analyze` כותב את `results/summary_table.md` ואת הגרפים מתוך
`results/*.json`. מדידות אמיתיות על המכונה מ-§1 (Qwen2.5-7B-Instruct, פרומפט
"short", 20 טוקני פלט):

| תצורה | קוונט׳ | TTFT (ש׳) | TPOT (מ״ש) | tok/s | שיא RAM (GB) | אנרגיה (Wh) | סטטוס |
|---|---|---|---|---|---|---|---|
| baseline (HF ישיר) | fp16 | — | — | — | — | — | **נכשל — נהרג ע״י ה-OS בטעינה (OOM)** |
| airllm | fp16 | 129.48 | 144,190 | 0.01 | 3.6 | 12.49 | תקין |
| ollama (GGUF) | q4 | 39.25 | 234.6 | **4.49** | 2.2 | 0.18 | תקין |
| ollama (GGUF) | q8 | 272.43 | 30,546 | 0.03 | 1.8 | 3.55 | תקין |

**קריאת הנתונים.**
- **קו הבסיס נכשל:** מודל ה-FP16 בן 15GB לא נכנס ל-8GB RAM ללא offload — ה-OS
  הורג את הטעינה. זהו צוואר הבקבוק של קיבולת הזיכרון (נשמר ב-`results/baseline_short.json`).
- **AirLLM מאפשר הרצה, אך לאט:** אותו מודל רץ ב-**3.6GB שיא RAM** בלבד ע״י זרימת
  שכבה-בכל-פעם מה-SSD — אך כל טוקן קורא מחדש את כל השכבות מהדיסק, ולכן הדיקוד
  ~**144 ש׳/טוקן** (0.01 tok/s). התנהגות חסומת-דיסק קלאסית.
- **הניצחון האמיתי — קוונטיזציה + כניסה ל-RAM:** **Q4 (~4.4GB) נכנס ל-RAM** →
  **4.49 tok/s** (פי ~600 מהר יותר מ-AirLLM) ב-0.18 Wh/בקשה.
- **צוק ה-RAM:** **Q8 (~8GB) *אינו* נכנס** → llama.cpp מדפדף אותו מהדיסק בכל טוקן
  → צונח ל-0.03 tok/s (פי ~130 איטי יותר מ-Q4). הגורם המכריע אינו רמת הקוונטיזציה
  עצמה אלא **האם מערך העבודה נכנס ל-RAM**.

> הערה: Ollama רץ בתהליך נפרד, ולכן "שיא RAM" שלו הוא דלתת RAM של כלל המערכת
> (דפדוף מבוסס-mmap גורם לדלתת Q8 להיראות קטנה) — הסתייגות מתועדת, לא סתירה.
> ראו `docs/PRD_quantization.md`.

## 4. ניתוח מושגי ההרצה (משימה 5.6)

- **Prefill = TTFT.** בניית ה-KV-cache על הפרומפט היא כפל-מטריצות מקבילי גדול →
  **חסום-חישוב (compute-bound)**. עקבו אחרי גידול ב-TTFT עם הפרומפט הארוך.
- **Decode = TPOT.** כל טוקן חדש מזרים את כל משקולות המודל דרך הזיכרון פעם אחת →
  **חסום-זיכרון (memory-bound)**. עם AirLLM המשקולות מגיעות מה-*SSD*, כך שה-Decode
  חסום ברוחב פס הקריאה מהדיסק — העלות הדומיננטית כאן.
- **AirLLM ≈ זיכרון וירטואלי / Paging** של מערכת הפעלה, אבל עבור *שכבות מודל*:
  רק ה"דף" (השכבה) הנחוץ נמצא בזיכרון, השאר על הדיסק. הדיסק הופך לצוואר הבקבוק
  האמיתי, לא גודל ה-RAM.
- **קוונטיזציה** מקטינה כל משקל (FP16→Q4 ≈ פי 4), מצמצמת בייטים לכל טוקן → דיקוד
  מהיר יותר וזיכרון שיא נמוך יותר, עד שהדיוק נפגע.

## 5. ניתוח כלכלי והמלצה (משימה 5.5)

![נקודת איזון](figures/break_even.png)

שני מודלי עלות שקופים ב-[`analysis/economics.py`](analysis/economics.py), כל
ההנחות ניתנות לעריכה ומצוינות:

- **API:** `בקשות × (קלט·מחיר_קלט + פלט·מחיר_פלט)`, עם הנחת **Prompt Caching**
  אופציונלית על הקידומת החוזרת (ספקים גובים פחות בהרבה על טוקנים מומכנים — זה
  *מזיז את נקודת האיזון ימינה*).
- **On-Prem:** CAPEX מופחת + חשמל (מתוך Wh/בקשה **שנמדד**) + תחזוקה.
- **ענן GPU (אופציונלי):** תעריף שעתי × שניות/בקשה.

**ממצא מחושב (הנחות ברירת המחדל שהוטמעו):** נקודת איזון ≈ **230 אלף בקשות**
בתקופת ההפחתה. מתחתיה ה-API זול יותר; מעליה On-Prem מנצח — *לפני* שקלול פרטיות,
אבטחת מידע וזמינות לא-מקוונת, שעשויים להטות לטובת On-Prem ללא קשר לנפח. **ערכו את
המחירים/התעריף/אורך החיים ב-`config/setup.json` (קטע `economics`) לערכים שאתם
מצטטים; האנרגיה לבקשה חייבת לבוא מההרצות שמדדתם.**

## 6. הרחבות (משימה 5.7)

ההרחבה המקורית הנדרשת בפרויקט היא **מסלול הקוונטיזציה ב-GGUF דרך Ollama** (§2.3):
הוא מאפשר השוואת Q4/Q8 אמיתית על GPU שאינו יכול להריץ bitsandbytes, ומנגיד שני
מנועי הסקה מקומיים שונים מהותית (זרימת שכבות מדיסק ב-AirLLM מול קוונטיזציה על CPU
ב-llama.cpp). הפרומפט `long_context` תומך בנוסף בסריקת אורך-קלט (TTFT מול אורך).

## שחזור

הכול רץ דרך ה-CLI `airllm-bench` (צרכן דק של מנשק ה-SDK, `AirLLMBenchSDK`),
מנוהל ב-uv. כל הפרמטרים הניתנים לכוונון נמצאים ב-`config/setup.json`.

```powershell
# 1. סביבה (uv בלבד). מוצג ב-PowerShell של Windows; bash אנלוגי.
uv venv
uv pip install -e .                      # כבד: torch, transformers, airllm, ...

# 2. תיעוד חומרה + צפייה במודל המומלץ (כבר מוטמע, אמיתי)
uv run airllm-bench hardware
uv run airllm-bench model

# 3. (אופציונלי) כיוון שכבות AirLLM לכונן עם מקום (ברירת מחדל .\layer_shards)
#    PowerShell:  $env:AIRLLM_SHARDS = "D:\airllm_shards"  (או ערכו config/setup.json)

# 4. הרצת הניסויים
uv run airllm-bench baseline             # טעינה נאיבית -> OOM צפוי = צוואר הבקבוק
uv run airllm-bench airllm               # זרימת שכבות FP16

# 5. השוואת קוונטיזציה דרך Ollama/GGUF (התקינו תחילה את אפליקציית Ollama)
uv pip install ollama
uv run airllm-bench ollama               # Q4 + Q8 על CPU

# 6. בניית טבלאות + גרפים + כלכלה
uv run airllm-bench analyze
#  ... או הכול ברצף אחד:
uv run airllm-bench all

# שערי איכות
uv run ruff check .
uv run pytest                            # 55 בדיקות, סף כיסוי >=85%
```

### מלכודות ידועות (מתוך רשימת ה-Do/Don't של ההגדרה)
- **Python 3.13 חדש מדי** ל-AirLLM/bitsandbytes — הפרויקט מצמיד `>=3.10,<3.13`
  (uv בוחר 3.12).
- **כוונו את `layer_shards_saving_path` לכונן עם מקום** — השכבות הן ~15GB של
  SafeTensors ל-7B ומציפות את `C:` אחרת. כאן פנויים רק ~40GB, אז עקבו אחרי הדיסק
  במהלך הרצת AirLLM.
- השתמשו ב-**`airllm.AutoModel`** (המחלקה הכללית) כדי להימנע משגיאת אי-התאמת
  מחלקה במודלים ממשפחת `Qwen`.
- **לעולם אל תשמרו את ה-Token של Hugging Face** — `.gitignore` מחריג טוקנים, את
  ה-venv, שכבות מודל, וקבצי `*.safetensors`/`*.gguf`.
- התחילו קטן (`max_new_tokens` נמוך) כדי לוודא שה"צינור" עובד, ואז הגדילו.
- **התאזרו בסבלנות:** על מכונה זו (8GB / SATA-SSD), AirLLM קורא ~15GB מהדיסק *לכל
  טוקן*; צפו לעשרות שניות לכל טוקן. שמרו `max_new_tokens` צנוע.
